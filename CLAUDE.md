# CLAUDE.md

本文件为 Claude Code 提供「健身大闯关」项目的上下文。完整设计与理由见同目录 `设计文档.md`。

## 项目概述

一个**单用户、本地优先的 PWA**,用于督促用户(frank)在**强直性脊柱炎安全范围内**减脂、改善体态、养成运动与饮食习惯。

核心机制:每天展示「今日该做什么」→ 每晚做**一次**复盘打卡(记录当天所做的事 + 带时间戳的照片)→ 据此**动态生成第二天的计划**。

目的是用工具督促行为改变,不是做一个功能复杂的产品。

## ⚠️ 最高优先级:健康安全约束

用户患有**强直性脊柱炎(AS,目前稳定)**和**轻度脊柱侧弯**,且**目前无专业运动指导**。任何涉及训练内容的代码或文案必须遵守:

- 训练内容默认走**保守档**,以活动度、伸展、温和强化为主。
- 避免脊柱大重量纵向负荷(大重量深蹲 / 硬拉 / 过头推举)、高冲击跑跳、壶铃摆动(swing)、传统屈曲卷腹。
- 计划必须**按当天症状分档(绿 / 黄 / 红)**调节强度;发作期只排温和活动度。
- App 必须内置医疗免责声明,并建议用户先做风湿科 / 康复科专业评估。
- 复盘表单含红旗信号自检,命中则次日 = 休息 + 就医建议。
- 详见 `设计文档.md` §2。

实现训练相关功能时,如不确定某动作是否安全,一律选择更保守的方案。

## 技术栈

- React + TypeScript + Vite + Tailwind CSS
- PWA(可添加到主屏)——**必须完全离线可用**(见下节)
- IndexedDB 本地存储(结构化数据 + 照片 blob)
- **无后端、无账号、无云**;单用户单设备 + JSON 导出备份

## 离线优先(硬要求)

App 必须在**完全无网络**时正常使用——这是产品的基本要求,不是可选项。

- 所有数据读写走本地 IndexedDB,运行时**不依赖任何网络请求**。
- 必须配置 **Service Worker**,预缓存 App Shell(HTML / JS / CSS / 字体 / 图标),首次加载后离线可完整打开。
- 不引入需要联网才能工作的运行时依赖(在线字体、CDN 脚本、远程 API);所需资源一律打包进构建产物。
- 动作示范图等静态资源随构建打包或存入本地缓存。
- 验收标准:断网后冷启动 App,今日 / 复盘 / 进展 / 动作详情均可正常使用。

## 架构关键决策

- **计划即数据**:训练计划是结构化 JSON(`planTemplate`),App 只负责渲染。调方案不改代码。
- **动态计划引擎**是透明的**纯函数规则引擎**,不是 AI 黑盒;必须可单元测试,并在 UI 上展示「为什么是今天这个安排」。
- **不自建推送通知**;提醒交给手机系统,App 只提供建议时间表。
- 照片只存本地,不上传(隐私)。

## 数据模型(IndexedDB)

- `profile / settings` — 身体信息、当前阶段、提醒时间、偏好
- `planTemplate` — 分期、训练模板 A/B、动作库(可整体替换的 JSON)
- `exercise` — 动作详情 + AS 安全等级 + 替代动作 + 累计完成次数(驱动详细度衰减)
- `dailyPlan` — 某天的计划:类型、动作列表、强度档、生成理由
- `checkIn` — 每天一条复盘记录(字段见设计文档附录 A)
- `photo` — 图片 blob + 时间戳 + 姿势标签(正 / 侧 / 背)
- `metrics` — 体重、腰围

## 动态计划引擎规则(核心逻辑)

输入:当前分期与本周已完成力量次数、昨晚复盘(症状档 / 完成度 / 疲劳)、连续训练天数、红旗信号。

症状分档:🟢 绿(晨僵 <30min,疼痛 ≤2)/ 🟡 黄(晨僵 30–60min 或疼痛 3–5 或明显疲劳)/ 🔴 红(晨僵 >60min 或疼痛 ≥6 或夜间痛)。

次日计划生成(按序判断,命中即停):

1. 红旗信号 → 休息 + 就医提示
2. 🔴 红 → 温和恢复日,无负荷
3. 🟡 黄 → 体态 / 活动度日(默认)或力量降档版
4. 🟢 绿 且 连续训练 ≥2 天 → 恢复 / 体态日
5. 🟢 绿 且 本周力量已达 3 次 → 体态 / 活动度日
6. 🟢 绿 其余 → 下一套力量训练(A/B 轮换)

附加:无论哪档,晚间体态放松每天都排(即「最低版本」)。详见 `设计文档.md` §5。

## 信息架构

底部 3 个 Tab:**今日** / **复盘** / **进展**;动作详情为子页面。

- 今日页:今日训练 + 晚间体态 + 饮食 / 喝水 + 「最低版本」兜底
- 复盘页:90 秒内可完成的每日打卡表单(每天仅一次,晚上),提交后展示明日计划预览
- 进展页:照片时间线对比 + 趋势(腰围 / 晨僵 / 完成率 / 连续天数)

## 测试与回归门禁(硬要求)

**每一次代码改动都必须带测试,不允许 regression。**

- 测试框架:Vitest(配合 Vite);React 组件用 React Testing Library。
- **动态计划引擎是测试重中之重**:每条规则(红旗、🔴 / 🟡 / 🟢 各分支、A/B 轮换、连续训练上限、晚间体态兜底)都必须有单元测试,覆盖边界与组合情况。
- 新增 / 修改功能时,**先写测试再写实现**(见下方准则 4)。
- 修 bug 时,先写一个能复现该 bug 的失败测试,再修到通过。
- **提交前必须本地跑通**:`npm run test` 全绿 + `npm run build` 成功。
- CI(GitHub Actions)在每次 push / PR 自动跑 lint + test + build,**失败不允许合并**——这是「不要 regression」的强制保证。

## 在 GitHub 上构建

- 项目托管在 GitHub,用 Git 管理;主分支(`main`)始终保持可构建、测试全绿。
- 功能开发走分支 + Pull Request;CI 通过后才合并到 `main`。
- `.github/workflows/ci.yml` 定义 CI 流程:install → lint → test → build。
- 部署:PWA 为纯静态产物,可用 GitHub Pages 托管,便于在手机上「添加到主屏」。
- 提交信息清晰,一次提交对应一件事。

## 版本管理

- 采用 SemVer,`package.json` 的 `version` 是唯一版本来源。里程碑对应:v0 = 0.1.0、v1 = 0.2.0、v2 = 0.3.0(已完成),v3 待规划。
- **每个 PR 合入 `main` 都要带版本信息**:功能 PR 递增 patch(如 0.2.0 → 0.2.1),里程碑完成递增 minor(如 → 0.3.0);版本号在 `package.json` 更新,并写进该 PR 的 commit / 合并信息(如 `feat: 数据导出备份 (v0.2.1)`)。
- 里程碑(v1 / v2 …)完成时打 git tag(如 `v0.3.0`)。
- App 内显示的版本号来自 `package.json`,构建时注入。

## 迭代顺序

先做能用的 MVP,不要一次做全;每个版本合并前 CI 必须全绿:

- **v0 MVP**:今日页(先用静态一周计划)+ 每晚复盘打卡(含时间戳照片)+ 本地存储 + Service Worker 离线
- **v1**:动态计划引擎 + 动作库可折叠详情(详细度衰减)+ 照片时间线
- **v2**:数据导出 / 导入备份 + 趋势图 / 周小结 + 版本号与更新提示 + 打卡提醒(系统 + App 内补提醒)+ 动作库(简笔示意图 + wger 补充,中英双语)

## 约定与 Non-goals

- 复盘表单必须能在 ≤90 秒内完成,多用快捷选项。
- 进步指标看腰围 / 照片 / 晨僵 / 完成率 / 连续打卡,**体重弱化**(只作月度参考)。
- 不做:点赞、评论、排行榜、社交、AI 滤镜、惩罚式打卡机制。
- 对「漏打卡」保持友好文案,不羞辱用户;允许补打卡(标注补录)。

## 通用编码准则(andrej-karpathy-skills)

> 以下 4 条改编自 [andrej-karpathy-skills](https://github.com/multica-ai/andrej-karpathy-skills)(MIT 许可),用于减少 LLM 常见编码错误。
> 取舍:这些准则偏向「谨慎优先于速度」;琐碎任务可自行判断。与上方项目专属规则冲突时,**以项目规则(尤其健康安全约束)为准**。

### 1. 想清楚再写(Think Before Coding)

**不要假设。不要藏着困惑。把权衡讲出来。** 动手前:

- 明确说出你的假设;不确定就问。
- 如果存在多种解读,把它们摆出来——不要默默选一个。
- 如果有更简单的做法,说出来;该反对时就反对。
- 有不清楚的地方,停下,指出困惑点,提问。

### 2. 简洁优先(Simplicity First)

**用解决问题的最小代码。不做投机性设计。**

- 不做没被要求的功能。
- 不为只用一次的代码做抽象。
- 不加没被要求的「灵活性」「可配置性」。
- 不为不可能发生的场景写错误处理。
- 如果写了 200 行而 50 行能搞定,重写。

自问:「资深工程师会不会觉得这过度复杂?」会,就简化。

### 3. 外科手术式改动(Surgical Changes)

**只动该动的。只清理自己制造的烂摊子。** 改动已有代码时:

- 不「顺手优化」相邻的代码、注释、格式。
- 不重构没坏的东西。
- 沿用既有风格,即使你会换个写法。
- 发现无关的死代码,提一句——不要删。

你的改动产生孤儿时:删掉因你的改动而不再被使用的 import / 变量 / 函数;不删除你来之前就存在的死代码,除非被要求。

准则:**每一行改动都应能直接追溯到用户的需求。**

### 4. 目标驱动执行(Goal-Driven Execution)

**定义成功标准,循环到验证通过为止。** 把任务转成可验证的目标:

- 「加校验」 → 「为非法输入写测试,再让它们通过」
- 「修 bug」 → 「写一个能复现 bug 的测试,再修到通过」
- 「重构 X」 → 「确保重构前后测试都通过」

多步任务,先给一段简短计划:

```
1. [步骤] → 验证:[检查点]
2. [步骤] → 验证:[检查点]
3. [步骤] → 验证:[检查点]
```

## 开发命令

- `npm install` — 安装依赖
- `npm run dev` — 本地开发(Vite)
- `npm run build` — 类型检查 + 生产构建(`tsc -b && vite build`)
- `npm run test` — 单元测试(`vitest run`,一次性;动态计划引擎规则必须有测试覆盖)
- `npm run test:watch` — 监听模式
- `npm run coverage` — 覆盖率(`vitest run --coverage`)
- `npm run lint` — 代码检查(`eslint .`)
- `npm run preview` — 预览生产构建(用于离线 / Service Worker 验证)
