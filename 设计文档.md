# 健身督促 App · 设计文档

> 版本:v0.5 · 2026-05-24(动作素材定案:简笔示意图必备、wger 作补充来源、动作内容中英双语)
> 用户:frank(单用户工具)
> 文档性质:产品设计 + 训练设计 + 健康安全设计
> 一句话定位:一个**存在手机本地的网页应用**,每天把"今天该做什么"讲清楚,每晚做**一次**复盘打卡(记录所做的事 + 带时间戳的照片),并据此**动态生成第二天的计划**。

---

## 文档目的

这份文档不是"为了做一个 app"。它的目的是:用一个尽量轻的工具,持续督促 frank 在**强直性脊柱炎安全范围内**减脂、改善体态、把运动和饮食变成稳定习惯。

工具只是手段。所以本文档同时写清楚三件事:**做什么(产品)**、**练什么(训练)**、**哪些不能碰(安全)**。三者冲突时,安全 > 训练 > 产品功能。

## 1. 背景与目标

### 1.1 用户画像

- 男,185cm / 83kg,BMI ≈ 24.3 —— 正常偏上,**并非超重**。
- 自述:肚子大、骨盆前倾、颈前倾;上肢力量弱(一个引体向上做不起来),纯健身新手。
- 真实身体状况判断:体重不高但体脂偏高、肌肉量偏低的"泡芙人"型,叠加久坐体态问题(上下交叉综合征)。肚子显大,有一部分来自骨盆前倾把小腹前推,而非纯脂肪。
- **健康状况(本设计的关键约束):**
  - 强直性脊柱炎(AS):目前**比较稳定、很少疼痛**。
  - 脊柱侧弯:**轻度,定期观察即可**。
  - 运动指导:**目前没有专业人士指导,完全自己摸索。**
- 器械条件:在家有哑铃、壶铃、瑜伽垫;但**器械可用性不固定** —— 在外 / 出差时可能纯徒手。方案不依赖任何杆。
- 时间:训练时间不固定 —— 可能在上午 / 下午 / 晚上,由用户**每次训练前选择**;需保证休息。
- 自评:坚持本身不是大问题,**缺的是清晰、可执行的方法**。

### 1.2 目标重构:为什么不是"减肥"

BMI 已属正常偏上。减脂和增肌同时发生时,体重秤几乎不动甚至略涨。如果把"掉秤"当目标,几周后秤不动就会误判为"没效果"而放弃——**这是这类计划最常见的失败原因**。

因此真实目标定为:

1. **减脂、缩小腰围、改善体态外观**;
2. 在 **AS 安全范围内**逐步增强力量、维持脊柱与髋关节活动度;
3. 养成**可持续**的运动与饮食习惯。

进步用这些指标衡量:腰围、同角度照片、晨僵时长、训练完成率、连续打卡天数。**体重只作为月度参考,不作为主指标。**

### 1.3 非目标(Non-goals)

- 不是多人产品,不上应用商店。
- 不自建推送通知系统(理由见 §8)。
- 不做点赞、评论、排行榜、社交、AI 滤镜。
- 不替代医疗:本工具不诊断、不治疗,不替代风湿科 / 康复科的意见。

## 2. 健康与安全约束【第一优先级】

> ⚠️ 本节优先级高于所有功能。任何与本节冲突的设计一律让步。

### 2.1 免责声明与前置建议

- 本工具及其中的训练 / 饮食内容仅供参考,**不构成医疗建议**。
- 鉴于用户当前**没有专业运动指导**,文档与 App 都将明确建议:**在开始负重训练前,先做一次风湿科或康复科 / 物理治疗师的评估**,确认可承受的运动类型与强度。在此之前,方案默认走**保守档**。
- App 首次启动需展示一次健康须知,并要求用户确认已知悉。

### 2.2 强直性脊柱炎(AS)运动原则

规律运动是 AS 公认最有效的非药物管理手段之一,**目标不是"练猛",而是维持脊柱 / 胸廓活动度、对抗向前弯曲僵直、控制炎症带来的不适**。

应优先做:

- 每日脊柱与髋关节**活动度 / 伸展**训练,尤其是**伸展(后伸)方向**——AS 倾向于把人固定成含胸驼背,训练要反向对抗。
- **深呼吸 / 扩胸**训练(AS 可能影响胸廓扩张)。
- **低冲击有氧**:快走、(若条件允许)游泳、骑行。游泳对 AS 尤其友好。
- 适度的**核心与背部稳定性**训练。
- 训练前充分热身;晨僵时温和活动有助缓解,但**不强行顶着锐痛练**。

需谨慎 / 暂时回避(在专业评估前):

- 脊柱大重量纵向负荷动作(大重量深蹲、硬拉、过头推举)。
- 高冲击、跑跳、有跌倒 / 撞击风险的运动。
- **壶铃摆动(swing)**等爆发 + 反复屈伸的动作——壶铃本身可作为配重(如高脚杯深蹲),但摆动类新手 + AS 谨慎。
- 传统卷腹 / 仰卧起坐等**反复脊柱屈曲**动作,用核心稳定类动作替代。
- 需要脊柱大幅扭转或屈曲下负重的动作。

### 2.3 脊柱侧弯(轻度)与体态

- 轻度侧弯:训练注重**双侧对称**、核心稳定与身体觉察,避免长期单侧过载;单侧动作两边组次对等。
- 骨盆前倾:髋屈肌 / 下背放松 + 臀肌、核心(抗伸展)强化。
- 颈前倾:收下巴(深层颈屈肌)+ 上背强化 + 胸部放松。

### 2.4 红旗信号(出现则立即停止并就医)

- 新发或明显加重的剧烈疼痛;
- 夜间痛明显、影响睡眠且持续;
- 疼痛或麻木**放射到腿部**;
- 关节明显肿胀、发热、伴发烧;
- 运动中胸背突发剧痛、或活动后异常无法缓解的疼痛。

App 的复盘表单中内置这些提示;命中时引导"明日计划=休息 + 就医建议"。

### 2.5 安全候选动作池

文档提供**安全候选动作池**而非铁律式处方;具体处方建议经康复科确认后再固定。

- **活动度 / 伸展**:温和猫牛、胸椎旋转、髋屈肌拉伸、腘绳肌拉伸、胸部开门拉伸、骨盆时钟、温和后伸(眼镜蛇 / 婴儿式交替)。
- **呼吸 / 扩胸**:扩胸深呼吸训练。
- **核心(稳定型,非屈曲)**:死虫、鸟狗、臀桥、温和平板、侧桥。
- **力量(中小负荷、躯干中立)**:高脚杯深蹲(中小重量)、哑铃单臂支撑划船、(上斜 / 跪姿到标准)俯卧撑、哑铃地面卧推、分腿蹲、轻负荷髋铰链(技术优先)。
- **姿势 / 后链**:温和 superman、YTW、靠墙天使。
- **有氧**:快走;有条件时游泳 / 骑行。

## 3. 设计原则

1. **安全优先** —— 见 §2,所有功能为其让步。
2. **降低决策成本** —— 打开 App 直接看到今天做什么,不需要思考、不需要翻计划。
3. **过程导向 > 结果导向** —— 打卡和反馈锚定"我有没有照做"(可控),不锚定体重(不可控)。
4. **永不归零** —— 每天都有一个"最低版本";状态差时做最低版,连续记录不断。
5. **计划是数据,App 是渲染器** —— 训练计划以结构化数据存在,调整方案不改代码。
6. **单用户、本地优先、隐私** —— 数据(含体态照片)只存在用户自己手机里,不上传。
7. **复盘不超过 90 秒** —— 每天唯一一次打卡,必须快,多用快捷选项,否则会被放弃。

## 4. 需求细化(功能需求)

### FR1 · 今日计划

- 首页只显示**今天**:今日训练(类型 + 动作清单 + 强度档)、晚间体态放松 routine、当日饮食指引、喝水目标。
- 今日计划由 **分期模板 + 昨晚复盘结果** 动态生成(见 §5)。
- 不显示未来若干天的计划,避免压迫感与"提前焦虑"。
- 每个训练日明确标出"最低版本"(状态差时的兜底内容)。
- 查看今日训练前,用户先选**场景**(时间段 + 有 / 无器械,见 §6.2);App 据此渲染对应的动作清单与提示。场景可重选。

### FR2 · 动作细化与详细度衰减

- 每个动作可展开**详情**:目标肌群、组数 / 次数 / 节奏、动作要领、常见错误、示意图、**AS 安全标注与替代动作**。
- **动作示意图(必备)**:每个动作都配一张简笔示意图(SVG 矢量,体积极小、随构建打包、离线可用),画出关键姿势与要领,降低"照着名字瞎做"的风险。这是动作详情的核心视觉,覆盖全部动作,由教练角色绘制提供。
- **wger 作补充来源**:开源动作库 wger(数据采用 CC-BY-SA 3.0)作为**补充** —— 当它有对应动作时,可引入其描述 / 图片丰富详情,须按许可署名;不整库导入,只取与本项目动作对应的条目,集成前对照 §2 做安全筛选。
- **中英双语**:动作的名称与文字说明保留中文 + 英文两份(英文可取自 wger 对应条目)。此处指动作内容双语,不含整个 App 界面的语言切换。
- **详细度自动衰减**:系统记录每个动作的累计完成次数。
  - 完成次数少(新手期)→ 详情默认展开。
  - 完成次数达到阈值 → 自动折叠为一行"动作名 + 组数次数",点击仍可展开。
- 用户可手动覆盖折叠 / 展开状态。
- 对应用户"前期需要、后面可能不用"的诉求。

### FR3 · 每晚复盘打卡(每天一次)

- 每天**仅一次**打卡,安排在晚上(默认提醒 21:30,可调)。
- 单次打卡即完成"统计当天做了什么 + 记录照片",字段见 **附录 A**,核心包括:
  - 今日训练完成情况(完成 / 部分 / 未做 / 今日为休息日);
  - 今日还做了什么(多选快捷标签:晚间体态、散步快走、喝够水、饮食克制、早睡等);
  - 饮食自评、喝水情况;
  - **AS 症状**:晨僵时长、疼痛 0–10、能量水平;
  - 心情;
  - 一句话备注(选填);
  - 照片(选填,自动带时间戳,见 FR5)。
- 提交后立即展示**明日计划卡预览**(见 FR4),形成"复盘→看到明天"的闭环。
- 漏打卡不做惩罚式设计;补打卡允许(标注为补录)。

### FR4 · 动态次日计划引擎

- 每次复盘提交后,引擎根据规则生成**第二天的计划**(详见 §5)。
- 引擎是**透明的规则引擎**,不是黑盒:计划卡上显示"为什么是今天这个安排"(例如:"昨日晨僵较久 → 今日为温和恢复日")。
- 用户可**手动覆盖**次日计划;当覆盖到比引擎建议**更高强度**时给出温和提示。

### FR5 · 照片时间线

- 复盘时可拍 / 选照片,系统**自动写入时间戳**;支持姿势标签(正面 / 侧面 / 背面)。
- 提供**时间线视图**:按时间排序浏览,可选两张并排对比。
- 拍照时叠加上一张同姿势照片的**半透明参考轮廓**,帮助对齐角度。
- 不做美颜 / 滤镜——诚实记录变化。
- 照片只存本地(IndexedDB)。

### FR6 · 数据与进度

- 周 / 月视图:腰围趋势、晨僵时长趋势、训练完成率、连续打卡天数;体重弱化展示(月趋势)。
- 每周生成一段简短**周复盘小结**(完成情况 + 一句鼓励 / 提示)。

### FR7 · 提醒(系统提醒 + App 内补提醒)

- App **不自建推送系统**(理由见 §8):纯本地、无后端的 PWA 无法在关闭状态下可靠推送(iOS 尤甚)。
- **系统提醒**:App 提供一个**提醒设置页**,给出建议时间表(训练提醒、复盘打卡提醒),引导用户一次性配置到手机自带"提醒事项 / 闹钟" —— 这是 App 关闭时唯一能准点响的方式。复盘提醒时间存入 settings,可调。
- **App 内补提醒**:用户打开 App 时,若已过设定的复盘时间且当天还没打卡,首页显著提示"今天还没打卡"。这是唯一能按"是否已打卡"做条件判断的提醒;系统提醒只能定点响、不判断。

### FR8 · 数据导出与备份

- 一键导出全部数据为单个文件(结构化数据 + 照片),用于换设备 / 防丢失。
- 支持导入恢复。
- App 内温和提示定期备份 —— 数据只在单设备本地,见 §13 风险。

### FR9 · 版本与更新

- App 内显示当前版本号(来源 `package.json` 的 `version`,构建时注入)。
- 检测到新版本时给非侵入式提示「有新版本,点击更新」,由用户触发更新,不再静默自动更新;离线时不检查更新、不影响使用。
- 版本规范见 `CLAUDE.md`:采用 SemVer,每个 PR 合入 `main` 时递增版本号、commit 信息带版本号,里程碑打 git tag。

## 5. 动态计划引擎(详细规则)

这是用户核心诉求"每晚复盘→动态决定第二天"的实现。对 AS 而言,这不只是激励设计,更是**按疾病波动调节负荷的必要机制**。

### 5.1 输入

- 当前分期与本周已完成力量训练次数;
- 昨晚复盘:**症状档位**、训练完成度、主观疲劳 / 能量;
- 距上次力量训练的天数、连续训练天数;
- 是否命中红旗信号。

### 5.2 症状分档

| 档位 | 判定(满足其一即归入更重档) | 含义 |
|------|------------------------------|------|
| 🟢 绿 | 晨僵 < 30min 或无;疼痛 ≤ 2/10;能量正常 | 状态稳定 |
| 🟡 黄 | 晨僵 30–60min;或疼痛 3–5/10;或明显疲劳 | 波动期 |
| 🔴 红 | 晨僵 > 60min;或疼痛 ≥ 6/10;或夜间痛 / 明显发作感 | 发作期 |

命中**红旗信号** → 直接覆盖为"休息 + 就医建议",优先级最高。

### 5.3 决策规则

按顺序判断,命中即停止:

1. **红旗信号** → 次日=休息 + 提示就医。
2. **🔴 红档** → 次日=温和恢复日:扩胸呼吸 + 温和活动度 + 短距离散步,**无任何负荷**。
3. **🟡 黄档** → 次日=体态 / 活动度日(默认);或力量降档版(组数 −1、重量下调、去掉强度较高动作)。引擎默认给体态日。
4. **🟢 绿档**,且**连续训练 ≥ 2 天** → 次日=恢复 / 体态日(强制保证休息)。
5. **🟢 绿档**,且**本周力量已达 3 次** → 次日=体态 / 活动度日或主动恢复。
6. **🟢 绿档**,其余情况 → 次日=下一套力量训练(A/B 轮换)。

附加规则:无论哪一档,**晚间体态放松 routine 每天都排**——它温和、助眠、对 AS 有益,是"永不归零"的最低版本。

### 5.4 透明与可覆盖

- 计划卡显示生成理由(如:"连续训练 2 天 → 今日安排恢复")。
- 用户可手动改;改到更高强度时提示一次。
- 规则先做简单版,后续用累积数据迭代。

## 6. 训练体系

### 6.1 分期

- **阶段 0 · 适应 / 评估期(第 1–2 周)**:以体态、呼吸、活动度为主;同时建立"每晚复盘"习惯;强度极低,目标是"别断"。建议此阶段去做一次专业评估。
- **阶段 1 · 基础期**:每周 3 次低-中强度全身训练(以哑铃中小重量、自重为主,避开脊柱大负荷),每日活动度 / 伸展。
- **阶段 2 · 进阶期**:在专业评估通过后,再考虑逐步加负荷与新动作。

### 6.2 训练时段与场景选择

训练时间不固定。用户**每次训练前选择一个「场景」**,App 据此渲染匹配的计划。场景含两个维度:

- **时间段**:上午 / 下午 / 晚上。影响热身与提示 —— 上午偏 AS 晨僵,热身更充分、起步更慢;晚上偏晚更温和,留睡前缓冲。**不改变动作清单。**
- **器械**:有器械(哑铃 / 壶铃可用,通常在家)/ 无器械(纯徒手)。**决定具体动作清单** —— 力量日含「有器械版」与「无器械(徒手)版」两套动作。

体态放松 routine 与复盘打卡仍固定在**晚上**进行(每日收尾 + 唯一一次打卡),与主训练在哪个时段无关。

最终计划 = 训练类型(静态轮换 / 动态引擎决定)× 场景(用户选的时间段 + 器械)。

### 6.3 A/B 训练轮换

因用户时间每天灵活,训练不绑定星期,而是排成 **A / B 两套全身训练轮换**:引擎判定"今天该力量训练"时,就取上次之后的下一套。每周凑够约 3 次即可。

### 6.4 示例一周(🟢 绿档顺利时)

仅为示意——实际由引擎按症状与完成度动态生成,不按固定星期。晚间体态放松与复盘每天进行,不在表中重复。

| 日 | 当天训练类型 |
|----|----|
| 1 | 力量 A |
| 2 | 活动度 / 体态 + 散步 |
| 3 | 力量 B |

## 7. 饮食与喝水

不给死板食谱,给一套"自己做饭"和"点外卖"都能用的原则(用户三餐为混合模式):

- **蛋白质优先**:泡芙人体重重组的第一杠杆,目标每天约 130–150g 蛋白质。自己做饭——每餐一个掌心大小的肉 / 蛋 / 豆制品;点外卖——选清蒸 / 卤 / 烤,主食减半,必要时加个鸡蛋或牛奶补蛋白。
- **餐盘法**:每餐大致 1/2 蔬菜、1/4 蛋白、1/4 主食;做饭点餐都适用。
- **抗炎倾向**:适当增加深海鱼 / omega-3、蔬果,减少高糖与高度加工食品。这对 AS 有一定益处,但**不夸大、不作为治疗手段**。
- **喝水** 2–2.5L/天,绑定既有习惯(起床、每餐、训练时各一杯)。
- 饮食与喝水**不单独多次打卡**,统一在每晚复盘里做一次自评。

## 8. 提醒策略

核心取舍:**不自建推送通知系统。**

网页应用无法稳定地在关闭状态下推送通知(尤其 iOS),为此搭推送服务器属于"为了 app 而 app"。因此:

- "提醒"这件事交给手机自带的提醒事项 / 闹钟,App 只提供建议时间表(见 FR7),引导用户**一次性配置**。
- 唯一的例外:App **打开时**可做一次"今天还没打卡"的补提醒(见 FR7)—— 条件提醒只能落在 App 内。
- App 自身专注两件事:**讲清楚今天做什么**、**让每晚那一次打卡足够顺手**。

## 10. 数据模型

本地存储,主要实体:

- **profile / settings**:身体信息、当前阶段、提醒时间、健康须知确认状态、上次备份时间、偏好。
- **planTemplate**:分期定义、训练模板(A / B,力量日含「有器械」与「无器械」两套动作)、动作库引用。可整体替换的 JSON。
- **exercise**:`id`、名称(中 / 英)、目标肌群、组数 / 次数 / 节奏、要领与常见错误(中 / 英)、AS 安全等级、替代动作、**简笔示意图(SVG)**、wger 来源引用(选填)、**累计完成次数**(驱动详细度衰减)。
- **dailyPlan**:`date`、类型(力量A / 力量B / 体态 / 恢复 / 温和)、动作列表、强度档、**生成理由**。
- **checkIn**(每天一条):`date`、训练完成度、完成事项标签、饮食自评、喝水、症状(晨僵分钟 / 疼痛 0–10 / 能量)、心情、备注、照片引用、是否补录。
- **photo**:`id`、图片 blob、拍摄时间戳、姿势标签(正 / 侧 / 背)、关联 checkIn。
- **metrics**:`date`、体重、腰围。

存储:全部使用 IndexedDB(照片 blob + 结构化数据),配 JSON 导出 / 导入做备份。

## 11. 技术架构

### 11.1 形态与技术栈

- **形态**:PWA,可"添加到主屏幕",全屏运行,不上应用商店。
- **技术栈**:React + TypeScript + Vite + Tailwind CSS。

### 11.2 离线能力(硬要求)

App 必须在**完全无网络**环境下正常使用——这是基本要求,不是可选项。

- 所有数据读写走本地 IndexedDB,运行时**不发起任何网络请求**。
- 配置 **Service Worker** 预缓存 App Shell(HTML / JS / CSS / 字体 / 图标);首次加载后离线可完整打开。
- 不引入需联网的运行时依赖(在线字体、CDN 脚本、远程 API);资源一律打包进构建产物。
- 验收:断网冷启动后,今日 / 复盘 / 进展 / 动作详情均可用。

### 11.3 存储

- IndexedDB,本地优先;**无后端、无账号、无云**。
- 理由:单用户、隐私(体态照片不出本机)、免费、零维护、离线可用。
- 代价:数据绑定单设备、不跨设备同步 → 用 FR8 的导出 / 导入备份兜底。

### 11.4 动态计划引擎与"计划即数据"

- 动态计划引擎为**纯前端规则函数**,透明、可解释、可单元测试。
- planTemplate 为可替换 JSON,由"教练角色"维护、按周调整,App 只负责渲染。
- 提醒依赖系统提醒事项,App 不做推送。

### 11.5 测试与回归门禁

- 测试框架:Vitest;React 组件用 React Testing Library。
- **每一次改动都必须带测试,不允许 regression。**
- **动态计划引擎(§5)是测试重中之重**:每条规则分支、边界与组合情况都要有单元测试。
- 开发模式:先写测试再写实现;修 bug 先写一个能复现的失败测试。
- 提交前本地跑通 `npm run test` + `npm run build`。

### 11.6 GitHub 托管与 CI

- 项目托管在 GitHub,用 Git 管理;`main` 分支始终保持可构建、测试全绿。
- 功能走分支 + Pull Request;`.github/workflows/ci.yml` 在每次 push / PR 自动跑 lint + test + build,**失败不允许合并**。
- 部署:纯静态产物,可用 GitHub Pages 托管,便于手机端"添加到主屏"。

### 11.7 编码准则

采用 andrej-karpathy-skills 的 4 条准则(想清楚再写 / 简洁优先 / 外科手术式改动 / 目标驱动执行),已并入 `CLAUDE.md`,供 Claude Code 编码时遵循。

## 13. 风险与开放问题

**风险**

- *最大风险:无专业评估即开始负重训练。* 缓解:方案默认保守档;App 内置免责与红旗提示;文档明确建议先做风湿科 / 康复科评估。
- *单设备数据丢失。* 缓解:FR8 导出备份;提示用户定期备份。
- *复盘疲劳——用户嫌每天填麻烦。* 缓解:复盘控制在 90 秒内,多用快捷标签;不做惩罚式设计。
- *规则引擎过简或过繁。* 缓解:先上简单规则,用累积数据迭代。

**开放问题**

1. 是否需要跨设备同步?(当前结论:暂不,单设备 + 备份。)
2. 晚间训练(8 点后)结束到入睡的缓冲是否充足?偏紧时把强度往前压。
3. 体态照片是否需要 App 内简单锁(隐私)?
4. planTemplate 由谁、多久更新一次?(建议:按周,由教练角色结合复盘数据。)
