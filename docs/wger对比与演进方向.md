# wger 对比研究与演进方向

> 文档性质:横向对标 + 版本演进参考
> 对标对象:wger(开源健身管理器)· 信息截至 2026-05
> 用途:厘清「我们该往哪走、哪些坚决不学」,作为 v3 及以后的方向参考

---

## 1. 为什么做这份对比

wger 是健身类开源软件里最成熟的项目之一(10+ 年、8600+ 提交、Web + 原生 App 全平台)。逐项对照它和我们的设计,不是为了照抄,而是:

- 用一个成熟样本反衬出我们**刻意做了哪些不同选择**,验证这些选择;
- 从它的成熟做法里,挑出**少数几项真正值得借鉴**的;
- 划出**红线** —— 它有、但我们一旦沾上就会丢掉自身价值的东西;
- 给未来的功能决策留**一把尺子**。

结论先行:wger 和我们是**两个不同物种**的健身 App,而且这是我们主动选的。

## 2. wger 速写

**定位**:免费、开源(AGPL-3.0)、可自托管、**多用户**的综合健身 / 营养 / 体重管理器,还兼做健身房管理工具。

**技术**:Django + Python 后端,Postgres + Redis + gunicorn,Docker 部署;移动端是 Flutter 原生 App(Android / iOS),带多端同步、条码扫描。

**功能面(大而全)**:

- 训练:自定义 routine、自动加重进阶规则、100+ 动作的动作 wiki;
- 营养:接 Open Food Facts(200 万+ 食物),记录卡路里与宏量;
- 追踪:体重、自定义体测项、照片进度画廊;
- 平台:完整 REST API、多用户、健身房管理、社区翻译(Weblate)。

**产品哲学**:wger 是一个**中立的容器** —— 它不预设"你今天该练什么",而是给你工具和数据库,你自己搭计划、自己记录。它的目标是**服务所有人、所有用例、长期可自托管**。灵活、全面,是它的核心竞争力。

## 3. 我们的 App 速写(对照)

**定位**:**单用户**、本地优先、离线 PWA;无后端、无账号、无云。

**核心**:一个**有观点的「教练」** —— 每天直接告诉 frank 今天做什么(动态引擎按昨晚症状生成),训练内容按**强直性脊柱炎安全约束**裁剪。目标是**行为改变**,不是数据管理。

**产品哲学**(见设计文档):目的不是做一个 app,是督促一个**具体的人**动起来;刻意极简(无社交、无排行榜);数据只在本机(隐私)。

## 4. 根本差异

| 维度 | wger | 我们(fitness-quest) |
|------|------|----------------------|
| 用户 | 多用户 | 单用户(frank 一个人) |
| 部署 | 自托管 / 服务器(Docker + Postgres + Redis) | 本地优先,无后端、无云 |
| 形态 | Web + Flutter 原生 App | PWA(加到主屏) |
| 立场 | 中立容器:你自己搭计划、自己记录 | 有观点的教练:每天告诉你做什么 |
| 范围 | 大而全:训练 + 营养 + 体重 + 体测 + 健身房管理 | 小而精:一个「看计划 → 打卡 → 生成明天」闭环 |
| 数据 | 服务器数据库,可多端同步 | IndexedDB 本机,导出文件备份 |
| 健康定制 | 通用,无特定医疗约束 | AS 安全是第一约束,症状分档驱动 |
| 目标 | 服务所有人、所有用例、长期 | 让这一个人真的动起来、用得久 |
| 成熟度 | 10+ 年、8600+ 提交、跨平台 | 新生,v1 完成 |

**解读**:差异不是"我们还没做到 wger 那样",而是**目标函数不同**。wger 优化的是"覆盖所有人 / 所有用例";我们优化的是"让 frank 这一个人真的去练、并用得久"。两边各自的设计,对各自的目标都是对的。

## 5. 可以向 wger 借鉴的(过闸后纳入未来版本)

挑选标准:和我们"单用户 / 本地 / 极简 / 有观点"的取向不冲突,且能增强"督促健身"。

1. **动作数据建模**。wger 的 exercise 模型(类别 / 器械 / 主次肌群 / 变式 / 多语言)很成熟。我们的 `exercise` 实体在做双语和 wger 补充时,可参考它的字段结构。低风险。
2. **灵活的体测项**。wger 支持"自定义测量项"。我们现在只有体重 + 腰围;若以后想跟踪更多(如颈围,对颈前倾有意义),一个可扩展的 measurement 模型比硬编码更干净。低风险、小改动。
3. **进阶 / 加重规则的建模**。wger 有"自动加重进阶规则"。等我们做到分期里的**阶段 2 进阶期**(设计文档 §6.1),它的规则建模值得参考 —— 但要包进我们的引擎、保持透明可测试。中等相关,时机在 v3 之后。
4. **导出格式的开放性**。wger 一切皆有 API、数据格式开放。我们的导出备份(FR8)也应做成**清晰、自描述的开放格式**(而非黑盒),利于长期备份与迁移。低风险,v2 PR1 就该注意。

## 6. 必须刻意不学的(红线)

这些是 wger 有、但我们一旦引入就会丢掉自身价值的:

- **多用户 / 账号 / 社交 / 健身房管理**。我们是单用户、隐私、无社交 —— 这是设计原则,不是"还没做"。
- **后端 / 服务器 / 自托管**。本地优先(无后端、离线、零维护、免费、照片不出本机)是核心架构决策。引入服务器,隐私和离线一起塌。
- **全量营养追踪**。wger 接 200 万食物库、记卡路里宏量。我们刻意把饮食压到"每晚复盘里一次自评" —— 因为每日多次记录会被放弃(设计文档 §7、复盘 ≤90 秒原则)。这条最容易让人手痒,务必守住。
- **「中立容器」这个定位本身**。wger 不替你决定练什么 —— 这正是它的灵活,也正是它的负担(你得自己会搭计划)。我们的全部价值在于**反过来**:替你决定、降低决策成本。一旦开始往"灵活、可配置、大而全"走,就没有存在的理由了。

一个反直觉的点:wger 再成熟,也**给不了** frank 我们最核心的东西 —— AS 安全约束 + 症状驱动的动态引擎。通用工具没法在医疗上"有立场"。我们的**专用性就是护城河**。

## 7. 一把尺子:新功能决策框架

每想加一个功能,先过这五问,任一不过就别做(或重新设计到能过):

1. 它让 frank **更可能真的去练**吗?(服务"督促",不是为了功能完整)
2. 它能**离线、数据留在本机**吗?
3. 它能保持**单用户、无后端**吗?
4. 它**不增加每日操作负担**吗?(复盘仍 ≤90 秒;不引入需每天多次记录的东西)
5. 资深工程师会觉得**过度复杂**吗?(Karpathy 简洁优先)

## 8. 演进方向建议

- **近期(v2 内)**:wger 仅作动作库的**补充数据源**(已定)。不碰它的任何其他部分。
- **中期(v3+)**:有限借鉴 §5 的四项 —— 动作数据建模、灵活体测、进阶规则建模、导出格式开放化。每项先过 §7 的尺子。
- **v3 本身**:设计文档已规划的"教练角色每周读打卡数据、自动调下周计划" —— 这个方向**符合我们的 DNA**(强化"督促"、不堆功能),优先级高于任何"向 wger 看齐"的想法。
- **长期红线**:永不引入多用户 / 后端 / 社交 / 全量营养追踪;不为"对标 wger"而堆功能。我们要成为的不是"另一个 wger",而是"**最适合 frank 一个人、能让他用得久的督促工具**"。

## 9. 一句话总结

wger 是一座设备齐全的**公共健身房**;我们要做的,是 frank 家里那位**只懂他、每天推他一把的私人教练**。健身房再大,也代替不了那个教练;而那个教练,也永远不该试图把自己变成一座健身房。
